Higher-order functions, functions as arguements.

Functions that manipulate functions are called higher-order functions, which take functions as arguments or return functions as return values.

  • Express general methods of computation

  • Remove repetition from programs

  • Separate concerns among functions

Design higher-order functions

Generalize Patterns with Arguments(which may be functions). It's about Generalization!

Finding common structure allows for shared implementation.

Our environment model of how to evaluate a call expression extends gracefully to the case of higher-order functions, without change. When a user-defined function is applied to some arguments, the formal parameters are bound to the values of those arguments (again, which may be functions) in a new local frame.

functions as arguments

See the example below.

def summation(n, turn):
    """sum up to n with turn(n)
    >>> summation(3, square) # 1 + 4 + 9 = 14
    14
    >>> summation(3, cube) # 1 + 8 + 27 = 36
    36
    """
    total, k = 0, 1
    while k <= n:
        total += turn(k)
        k += 1
    return total

def square(x):
    return x*x

def cube(x):
    return x*x*x

Now we begin to see a more powerful kind of abstraction: some functions express general methods of computation, independent of the particular functions they call.

There is great example named golden ratio.

The [golden ratio, often called "phi"(ϕ\phiϕ), is a number near 1.6 that appears frequently in nature, art, and architecture.

Among the well-known properties of the golden ratio, there are two:

  • it can be computed by repeatedly summing the inverse of any positive number with 1
  • it is one less than its square.

ϕn=1ϕn−1+1,  ϕ1=1ϕ2−1=ϕ\begin{aligned} \phi_n=\frac{1}{\phi_{n-1}}+1,\ \ \phi_1=1 \\ \phi^2 -1 = \phi \end{aligned} ϕn​=ϕn−1​1​+1,  ϕ1​=1ϕ2−1=ϕ​

And it also has an exact closed-form solution.

ϕ=1+52≈1.618\phi = \frac{1+\sqrt 5}2\approx 1.618 ϕ=21+5​​≈1.618

We can create a improve function to do this kind of iterative improvement algorithm. Begin with a guess of a solution to an equation. It repeatedly applies an update function to improve that guess, and applies a close comparison to check whether the current guess is "close enough" to be considered correct.

def improve(update, close, guess=1):
    while not close(guess):
        guess = update(guess)
    return guess

According to the properties above, we can define the golden_update and golden_close functions.

def golden_update(x):
    return 1 / x +1

def golden_close(x):
    return close_enough(x, x**2-1)

Then we should define a close_enough fucntion to check if the value after iteration is close enough to the true value. We can use the arguments tolerance to decide whether two value is close enough.

def close_enough(x, y, tolerance=1e-15):
    return abs(x - y) < tolerance

We can check if it's close enough to the true value.

>>> phi_iterate = improve(golden_update, golden_close)
>>> phi = (1 + math.sqrt(5)) / 2
>>> assert close_enough(phi, phi_iterate)

For this test, no news is good news: Python returns None after its assert statement is executed successfully.

Nested definitions

Each general concept or equation maps onto its own short function. But it brings some negative consequences

  • The global frame becomes cluttered with names of small functions, which must all be unique.
  • Another problem is that we are constrained by particular function signatures: the update argument to improve must take exactly one argument.

Nested function definitions address both of these problems, but require us to enrich our environment model.

The solution to both of these issues is to place function definitions inside the body of other definitions. Like local assignment, local def statements only affect the current local frame.

Let's consider a new problem: computing the square root of a number. In programming languages, "square root" is often abbreviated as sqrt. There are some properties too:

(a)n=avg[(a)n−1, a(a)n−1](\sqrt a)_n = {\rm avg} \left [(\sqrt a)_{n-1},\ \frac{a}{(\sqrt a)_{n-1}} \right] (a​)n​=avg[(a​)n−1​, (a​)n−1​a​]

Let's see how can we do this iteration with function improve and close_enough using nested definition:

def avg(x, y):
    return (x+y) / 2
def sqrt(a):
	def update(x):
		return avg(x, a/x)
    def close(x):
        return close_enough(x * x, a)
    return improve(update, close)

We can use names like update and close which are more general by using nested definition.

By the way, the assert test goes right.

>>> assert close_enough(sqrt(10), math.sqrt(10))

Like local assignment, local def statements only affect the current local frame. These functions are only in scope while sqrt is being evaluated.

Lexical scope. Locally defined functions also have access to the name bindings in the scope in which they are defined.

In this example, update refers to the name a, which is a formal parameter of its enclosing function sqrt. This discipline of sharing names among nested definitions is called lexical scoping. Critically, the inner functions have access to the names in the environment where they are defined (not where they are called).

We require two extensions to our environment model to enable lexical scoping.

  1. Each user-defined function has a parent environment: the environment in which it was defined.
  2. When a user-defined function is called, its local frame extends its parent environment.

Functions defined within other function bodies are bound to names in a local frame.

Function values each have a new annotation that we will include in environment diagrams from now on, a parent.

The parent of a function value is the first frame of the environment in which that function was defined.

When a user-defined function is called, the frame created has the same parent as that function.

Extended Environments. An environment can consist of an arbitrarily long chain of frames, which always concludes with the global frame.

Hence, we realize two key advantages of lexical scoping in Python.

  • The names of a local function do not interfere with names external to the function in which it is defined, because the local function name will be bound in the current local environment in which it was defined, rather than the global environment.
  • A local function can access the environment of the enclosing function, because the body of the local function is evaluated in an environment that extends the evaluation environment in which it was defined.
